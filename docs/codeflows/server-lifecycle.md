# Server Lifecycle (ur start / ur stop)

## Overview

`ur start` brings up the full stack: builderd (host daemon) + Docker Compose services
(ur-postgres, ur-server, ur-squid). `ur stop` tears everything down in reverse.

## Port Allocation

| Service | Default Port | Config Field | Derivation |
|---------|-------------|-------------|------------|
| ur-server (gRPC) | 12321 | `server_port` | Explicit or default |
| Worker gRPC | 12322 | `worker_port` | `server_port + 1` |
| builderd (gRPC) | 12323 | `builderd_port` | `server_port + 2` |

All ports derive from `server_port` when not explicitly set, ensuring test isolation
when using a custom `server_port`.

## Compose Generation

`generate_compose()` in `crates/ur/src/compose.rs` builds YAML programmatically from
`ur.toml` config (network names, container names, proxy hostname).
No static template вЂ” the compose file is generated fresh on every `ur start`.

Environment variables passed to `docker compose`:
- `UR_CONFIG` вЂ” host config directory (mounted as `/config` in server)
- `UR_WORKSPACE` вЂ” host workspace directory (mounted as `/workspace` in server)
- `UR_SERVER_PORT` вЂ” gRPC listen port for ur-server
- `UR_BUILDERD_PORT` вЂ” builderd port (server uses this to build `UR_BUILDERD_ADDR`)
- `DATABASE_URL` вЂ” Postgres connection URL for ur-server (e.g., `postgres://ur:ur@ur-postgres:5432/ur`)

The server container receives `UR_BUILDERD_ADDR=http://host.docker.internal:$UR_BUILDERD_PORT`
to reach builderd on the host via Docker's host gateway.

The `ur-postgres` container stores data at `$UR_CONFIG/postgres` (mounted as `/var/lib/postgresql/data`).
If backup is configured, the host backup path is mounted at `/backup` in the postgres container for `pg_dump`/`pg_restore`.

### Postgres Bootstrap Databases

On the very first start (empty data volume), the Postgres container runs init scripts from
`$UR_CONFIG/postgres-init/` (mounted at `/docker-entrypoint-initdb.d` inside the container).
The file `01_create_databases.sql` is auto-generated by `compose_manager_from_config()` and creates:

- `ur_tickets` вЂ” logical database for all ticket, edge, activity, and slot data
- `ur_workflow` вЂ” logical database for workflow state and events

Both databases are owned by the configured Postgres user (default: `ur`).
`POSTGRES_DB` is set to the user name (`ur`) as the bootstrap database the server connects to
on startup вЂ” the application databases are created by the init script, not `POSTGRES_DB`.

The init script uses `SELECT ... WHERE NOT EXISTS` guards so it is safe to run against an
already-initialized cluster (though Docker only executes `/docker-entrypoint-initdb.d` scripts
when the data directory is empty).

## Concurrency Safety

- `start_builderd()` uses an exclusive file lock (`builderd.lock`) to prevent races
  between concurrent `ur start` invocations
- PID file checked under lock вЂ” stale PIDs are cleaned up before spawning

## Key Files

- `crates/ur/src/builderd.rs` вЂ” builderd lifecycle (start/stop, PID management)
- `crates/ur/src/compose.rs` вЂ” ComposeManager + `generate_compose()`
- `crates/ur/src/main.rs` вЂ” `start_server()` / `stop_server()` orchestration
- `crates/ur_config/src/lib.rs` вЂ” port defaults, config parsing
