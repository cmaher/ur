# Database Lifecycle

How the Postgres databases are initialized, migrated, queried, backed up, and shut down in the two-DB architecture.

## Overview

The system uses **two separate Postgres databases** managed by sqlx with async access via `PgPool`, automatic per-crate migrations, and optional periodic backups via `pg_dump` into the `ur-postgres` container.

| Database | Default name | Crate | Domain |
|----------|-------------|-------|--------|
| Ticket DB | `ur_tickets` | `ticket_db` | tickets, edges, activities, metadata, slots, workers, UI events |
| Workflow DB | `ur_workflow` | `workflow_db` | workflow state, events, intents, comments, UI events |

Both databases live in the same `ur-postgres` container but have independent connection pools, independent migration histories, and independent backup schedules.

## Connection URL Resolution

The server connects to each database via its own URL.

```
init_ticket_database(cfg)
  → std::env::var("TICKET_DATABASE_URL")
    → falls back to cfg.ticket_db.database_url()
      → DatabaseManager::open(&url)   [ticket_db pool]

init_workflow_database(cfg)
  → std::env::var("WORKFLOW_DATABASE_URL")
    → falls back to cfg.workflow_db.database_url()
      → DatabaseManager::open(&url)   [workflow_db pool]
```

`TicketDbConfig` and `WorkflowDbConfig` in `ur_config` each hold host, port, user, password, and name fields with a `database_url()` method.

- **Defaults** (both): host=`ur-postgres`, port=`5432`, user=`ur`
- **Default names**: `ur_tickets` (ticket DB), `ur_workflow` (workflow DB)
- **Password override**: `UR_TICKET_DB_PASSWORD` env var for the ticket DB; `UR_WORKFLOW_DB_PASSWORD` for the workflow DB

## Postgres Bootstrap Databases

On a fresh volume the Postgres container runs `$UR_CONFIG/postgres-init/01_create_databases.sql`
(auto-generated by `compose_manager_from_config()` and mounted at `/docker-entrypoint-initdb.d`).
This creates both logical databases before `ur-server` runs its migrations:

- `ur_tickets` — ticket, edge, activity, slot, and worker tables
- `ur_workflow` — workflow, workflow_event, workflow_intent, and related tables

`POSTGRES_DB` (the bootstrap database Postgres creates when the cluster is initialized) is set to
the user name (`ur`). This is a system-level database; application code always connects to one of
the two above.

## Initialization and Migration

On startup (`crates/server/src/main.rs`), each database goes through the same initialization flow independently:

1. `DatabaseManager::open(url)` creates an sqlx `PgPool`
2. `sqlx::migrate!().run(&pool)` applies pending migrations from the crate's `migrations/` directory
3. Migrations are embedded at compile time via the `sqlx::migrate!()` macro

Each crate owns its own migration history:

| Crate | Migration path | Notes |
|-------|---------------|-------|
| `ticket_db` | `crates/ticket_db/migrations/` | ticket, edge, meta, activity, slot, worker, ui_events, triggers |
| `workflow_db` | `crates/workflow_db/migrations/` | workflow, workflow_event, intent, comments, ui_events, triggers |

**Never modify existing migration files** — sqlx checksums applied migrations and will fail if they change. Always add a new numbered migration file for schema changes.

## Schema

### ticket_db (ur_tickets)

```
ticket     (id PK, type, status, priority, parent_id FK→ticket, title, body, created_at, updated_at, project, lifecycle_status, branch, lifecycle_managed)
edge       (source_id FK, target_id FK, kind) — PK: (source_id, target_id, kind)
meta       (entity_id, entity_type, key, value) — PK: (entity_id, entity_type, key)
activity   (id PK, ticket_id FK, timestamp, author, message)
slot       (id PK, project_key, slot_name, host_path, created_at, updated_at)
worker     (worker_id PK, process_id, project_key, container_id, worker_secret, strategy, container_status, agent_status, workspace_path, created_at, updated_at, idle_redispatch_count)
worker_slot (worker_id FK, slot_id FK) — PK: (worker_id, slot_id)
ticket_comments (comment_id, ticket_id FK, pr_number, gh_repo, reply_posted, created_at) — PK: (comment_id, ticket_id)
ui_events  (id BIGSERIAL PK, entity_type, entity_id, created_at) — ephemeral event buffer
```

### workflow_db (ur_workflow)

```
workflow          (id PK, ticket_id, status, created_at, stalled, stall_reason, implement_cycles, worker_id, noverify, feedback_mode, ci_status, mergeable, review_status)
workflow_event    (id PK, ticket_id, old_lifecycle_status, new_lifecycle_status, attempts, created_at)
workflow_intent   (id PK, ticket_id, target_status, created_at)
workflow_comments (ticket_id, comment_id) — PK: (ticket_id, comment_id)
workflow_events   (id PK, workflow_id, event, created_at)
ui_events         (id BIGSERIAL PK, entity_type, entity_id, created_at) — ephemeral event buffer
```

The `ui_events` table exists in **both** databases. Each database has its own copy with its own triggers. The server runs two `PgEventPoller` instances and merges their streams before dispatching to gRPC subscribers (see `docs/codeflows/ui-events.md`).

The `workflow` table references `ticket_id` as a plain TEXT column — there is no foreign-key constraint across databases. This is an intentional **soft reference**: the application layer enforces referential integrity.

## Component Interactions

```
server main.rs
  │
  ├── ticket_db::DatabaseManager::open(ticket_url)
  │     └── ticket_pool (PgPool)
  │           ├── GraphManager(ticket_pool)
  │           │     └── builds petgraph DiGraph from edge table
  │           ├── TicketRepo(ticket_pool, GraphManager)
  │           │     └── CRUD: tickets, activities, metadata, edges
  │           ├── WorkerRepo(ticket_pool)
  │           │     └── worker and slot management
  │           ├── UiEventRepo(ticket_pool)
  │           │     └── poll and delete ui_events rows (ticket_db copy)
  │           └── SnapshotManager(container_command, container_name, "ur_tickets")
  │                 └── pg_dump / pg_restore for the ticket DB
  │
  └── workflow_db::DatabaseManager::open(workflow_url)
        └── workflow_pool (PgPool)
              ├── WorkflowRepo(workflow_pool)
              │     └── workflow lifecycle, events, intents, comments
              ├── UiEventRepo(workflow_pool)
              │     └── poll and delete ui_events rows (workflow_db copy)
              └── SnapshotManager(container_command, container_name, "ur_workflow")
                    └── pg_dump / pg_restore for the workflow DB
```

### Wiring in main.rs

```rust
let ticket_db  = ticket_db::DatabaseManager::open(&ticket_url).await?;
let workflow_db = workflow_db::DatabaseManager::open(&workflow_url).await?;

// Ticket subsystem
let graph_manager = GraphManager::new(ticket_db.pool().clone());
let ticket_repo   = TicketRepo::new(ticket_db.pool().clone(), graph_manager);
let worker_repo   = WorkerRepo::new(ticket_db.pool().clone());

// Workflow subsystem
let workflow_repo = WorkflowRepo::new(workflow_db.pool().clone());

// UI events — two pollers, one merged stream (see ui-events.md)
let ticket_ui_poller   = PgEventPoller::new(ticket_db.pool().clone(), ...);
let workflow_ui_poller = PgEventPoller::new(workflow_db.pool().clone(), ...);

// Backup subsystem — one BackupTaskManager per database
let ticket_snapshot   = SnapshotManager::new(cmd.clone(), container_name.clone(), "ur_tickets");
let ticket_backup     = BackupTaskManager::new(ticket_snapshot, cfg.ticket_db.backup.clone());
ticket_backup.spawn(shutdown_rx.clone())?;

let workflow_snapshot = SnapshotManager::new(cmd, container_name, "ur_workflow");
let workflow_backup   = BackupTaskManager::new(workflow_snapshot, cfg.workflow_db.backup.clone());
workflow_backup.spawn(shutdown_rx.clone())?;
```

All managers receive pools via constructor injection. `DatabaseManager` is not passed around — only its pool is cloned and distributed.

## TicketRepo Queries

`TicketRepo` is the primary ticket data access layer, exposed via `TicketServiceHandler` over gRPC.

| Method | Operation |
|--------|-----------|
| `create_ticket` | INSERT into ticket |
| `get_ticket` | SELECT by id |
| `update_ticket` | Fetch-then-UPDATE (partial update via TicketUpdate) |
| `list_tickets` | Dynamic WHERE from TicketFilter (status, type, parent_id) |
| `set_meta` / `get_meta` / `delete_meta` | UPSERT/SELECT/DELETE on meta table |
| `add_edge` / `remove_edge` / `edges_for` | INSERT ON CONFLICT DO NOTHING/DELETE/SELECT on edge table |
| `add_activity` / `get_activities` | INSERT/SELECT on activity table |
| `tickets_by_metadata` / `tickets_with_metadata_key` | JOIN ticket+meta for metadata queries |
| `dispatchable_tickets` | Open children of epic with no open transitive blockers |
| `epic_all_children_closed` | COUNT non-closed children |

## Cross-DB References

The `workflow` table in `workflow_db` stores a `ticket_id` that logically refers to a row in `ticket_db.ticket`. Because the two pools connect to different Postgres databases, there is **no database-level foreign key** enforcing this relationship.

The application layer is responsible for consistency:

- Workflow records are only created after the ticket exists in `ticket_db`.
- Workflow handlers look up the ticket via `TicketRepo` (ticket pool) before acting on the workflow (workflow pool).
- If a ticket is deleted, its associated workflow record becomes an orphan; no cascade delete exists.

## Periodic Backup Flow

The backup architecture runs two independent `BackupTaskManager` instances — one per database. Each follows the same lifecycle.

The backup path on the host is mounted at `/backup` in the `ur-postgres` container via Docker Compose.

### Startup

1. `BackupTaskManager::new(snapshot_manager, backup_config)` — stores config and snapshot manager
2. `backup_task_manager.spawn(shutdown_rx)` — validates path, spawns tokio task
   - Returns `None` if `path` is not configured (backup disabled)
   - Returns `None` if `enabled = false`
   - Returns `None` (with warning) if path doesn't exist, isn't a directory, or isn't writable

### Backup Loop

```
loop {
    tokio::select! {
        sleep(interval) => {
            1. Generate timestamped filename: ur-{db_name}-YYYYMMDDTHHMMSSz.pgdump
            2. SnapshotManager::dump_to(filename)
               → docker exec ur-postgres pg_dump -Fc -f /backup/<filename> <db_name>
            3. On success: clean_old_backups() removes older ur-{db_name}-*.pgdump files
               (keeps retain_count most recent)
            4. On failure: log error, continue loop
        }
        shutdown_rx.changed() => {
            if shutdown signaled: run final backup, then return
        }
    }
}
```

### Restore

`SnapshotManager::restore_from(filename)`:
1. Runs `docker exec ur-postgres pg_restore --clean --if-exists -d <db_name> /backup/<filename>`
2. Restores directly into the live database, replacing existing data

### Manual Backup

`BackupTaskManager::run_once()` creates a manual backup with `manual-` prefix. Manual backups are excluded from automatic retention cleanup.

## Shutdown

In `main.rs`, after the gRPC server exits:

```rust
let _ = shutdown_tx.send(true);       // signal both backup tasks to stop

if let Some(handle) = ticket_backup_handle {
    let _ = handle.await;             // wait for ticket DB final backup
}
if let Some(handle) = workflow_backup_handle {
    let _ = handle.await;             // wait for workflow DB final backup
}
```

Both `PgPool`s are dropped when their `DatabaseManager` goes out of scope, closing all connections. Ticket pool is dropped before workflow pool (declaration order).
